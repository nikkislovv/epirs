# Флоу работы — от заявки до SPEC.md

> Единый алгоритм для всех шаблонов (vizitka, landing, portfolio).
> Реализация отличается, но путь от клиента до `SPEC.md` — всегда одинаковый.

---

## Шаг 0 — Определить входные данные

После первого контакта с клиентом выяснить два вопроса:

| Вопрос | Варианты |
|--------|----------|
| Есть ли ТЗ или дополнительные пожелания? | Нет / Да |
| Есть ли дизайн-материалы? | Нет / Figma / Скриншоты или макеты / Сайт-референс |

Это даёт **4 сценария**:

|  | Нет дизайна | Есть дизайн |
|--|-------------|-------------|
| **Нет ТЗ** | Сценарий 1 — дефолт | Сценарий 3 |
| **Есть ТЗ** | Сценарий 2 | Сценарий 4 |

---

## Шаг 1 — Приём заявки + сбор материалов (всегда)

Отправить клиенту:
1. `docs/IN_SCOPE.md` шаблона — чтобы подтвердил, что всё понял до оплаты
2. Google Form с вопросами (составлена по `docs/BRIEF.md`)

Получить от клиента:
- Заполненную форму
- ТЗ / дополнительные пожелания (если есть)
- Ссылку на Figma / скриншоты / сайт-референс (если есть)
- Тексты для всех секций
- Фото / изображения проектов
- Логотип (файл PNG/SVG или «текстовое лого»)
- Настройки Web3Forms (если нужна форма)
- Данные домена


**Задержка материалов = задержка дедлайна 1:1** — зафиксировать письменно.

---

## Шаг 2 — Анализ скоупа

**Если ТЗ нет** → пропустить, форма укладывается в стандартный тариф.

**Если ТЗ есть** → запустить агент `docs/agents/SCOPE_ANALYSIS.md`:
- Вставить содержимое ТЗ в промпт
- Агент выдаст: что в тарифе / что доплата / оценку часов
- Согласовать с клиентом состав и доп. стоимость до оплаты

---

## Шаг 3 — Определение стиля (в случае если нет исходных материалов дизайнов)

**Если есть референс (Figma design/Скриншоты/Ссылка на сайт референс)** → пропустить, дизайн формировать не нужно.

Запустить агент `docs/agents/STYLE_DETECT.md`.

**После ответа:** отправить клиенту 2-3 примера + предложенные цвета → получить письменный апрув.

Клиент **письменно подтверждает** стиль. Только после этого → Шаг 4.

---

## Шаг 4 — Развёртывание клиентского репо

Перед запуском AI_FLOW нужно создать чистый репозиторий клиента из шаблона и разложить все собранные материалы по нужным местам.

### 4.1 Создать репозиторий клиента

Создать новый git репозиторий и скопировать из шаблона (`vizitka/`) следующие файлы и папки:

```
src/                  ← весь boilerplate
docs/                 ← AI_FLOW.md, DEV_FLOW.md, agents/, BRIEF.md, IN_SCOPE.md
.claude/              ← skills/ui-ux-pro-max/
CLAUDE.md
index.html
package.json
vite.config.ts
tailwind.config.ts
tsconfig.json
postcss.config.js
.gitignore
```

### 4.2 Заполнить BRIEF.md

Перенести ответы клиента из Google Form в `docs/BRIEF.md`:
- Заполнить все секции по ответам из формы
- Раздел «Источник требований»: отметить есть ли ТЗ
- Раздел «Источник дизайна»: указать выбранный вариант
  - Если ветка A (нет дизайна): вписать URL референса, выбранного клиентом после STYLE_DETECT

### 4.3 Создать папку `customers-docs/`

Создать в корне клиентского репо папку `customers-docs/` и поместить туда:

```
customers-docs/
  TZ.md          ← если есть ТЗ от клиента (скопировать текст или вставить)
  layouts/       ← если есть скриншоты или макеты (любые форматы изображений)
```

Если ТЗ нет и дизайн-материалов нет — папка создаётся пустой (AI_FLOW проверит наличие файлов).

### 4.4 Установить зависимости

```bash
npm install
```

---

## Шаг 5 — Запустить AI_FLOW для создания SPEC.md

Открыть Claude Code в корне клиентского репо и следовать инструкциям из `docs/AI_FLOW.md`.

К этому моменту должно быть готово:
- [ ] `docs/BRIEF.md` заполнен
- [ ] `customers-docs/TZ.md` создан (если есть ТЗ)
- [ ] `customers-docs/layouts/` заполнен (если есть макеты)
- [ ] Источник дизайна определён и зафиксирован в BRIEF.md

---

## Шаг 6 — После создания SPEC.md переходить к реализации

Запустить Claude Code в корне клиентского репо:
- Claude читает `SPEC.md` → `CLAUDE.md` → начинает реализацию
- Стиль определён, контент готов, скоуп согласован
